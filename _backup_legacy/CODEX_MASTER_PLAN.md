## 1. 現状診断

### 1.1 技術スタック現状

| 項目 | 現状 |
|------|------|
| フレームワーク | Astro 5.13.9（静的生成、`is:inline`による部分ハイドレーション） |
| スタイリング | Tailwind CSS v4 + global.css（2,343行） |
| 3D | Three.js v0.182.0（バニラJS、260棟のインスタンスドメッシュ） |
| アニメーション | CSS Transitions + IntersectionObserver + RAF（ライブラリなし） |
| フォント | Barlow Condensed（Display） + Noto Sans JP（Body） |
| 依存関係 | わずか4パッケージ（Astro, Tailwind, @tailwindcss/vite, Three.js） |
| デプロイ | GitHub Pages（静的HTML） |

### 1.2 問題点の優先度順位

#### P0（致命的 - サイトの格を決定づけている）

| # | 問題 | 根拠 | 影響度 |
|---|------|------|--------|
| 1 | **Hero の第一印象が弱い** | MIT研究：人は50msで美的判断を下す。現在のHeroは「背景画像+テキスト」という標準構成で、AI建築事務所の異質さが伝わらない。`MOVE HEARTS. — 心を動かせ`の改行バランス・サイズ比率が中途半端（英語clamp(3.2rem,10.2vw,8.5rem) vs 日本語clamp(2rem,5.6vw,4.6rem)の比率が不統一）。 | 訪問者の70%はHeroで離脱判断する |
| 2 | **3D体験（ImmersiveEngine）が「すごい」に達していない** | 260棟のランダム配置ビルは建築の専門性を感じさせない。自動サイクル3.6秒は速すぎて内容を読めない。ポインタートラッキングは楽しいが目的が不明。Three.jsの563KB分の価値を提供できていない。 | 技術力を証明すべき箇所が逆効果 |
| 3 | **Signature Experience（企画から運用まで）がダサい** | 6フェーズのタブ切り替えは情報設計として正しいが、ビジュアル表現が事務的。左パネル+右タブ構成はSaaSダッシュボードのようで建築の空気感がない。ポインタートラッキンググラデーションは微弱すぎて効果を感じない。 | 会社の最大の差別化ポイントが埋没 |

#### P1（重大 - ユーザー体験を損なっている）

| # | 問題 | 根拠 |
|---|------|------|
| 4 | **ヘッダー導線が散漫** | primaryNavItems: 実績, AIプロダクト, お問い合わせ の3項目は少ないが、モバイルメニュー展開時のfooterNavItemsは多すぎる。デスクトップでのロゴ表示が420px以下で消える実装は不自然。 |
| 5 | **セクション間の呼吸がない** | py-20(80px)〜py-28(112px)の垂直余白はラグジュアリーサイトの基準（160-200px）に対して圧倒的に狭い。コンテンツが密集し「情報サイト」の印象を与える。 |
| 6 | **フォント選定がブランドに不一致** | Barlow Condensedは工業系・スポーツ系のCondensedフォントで、建築の重厚さ・知性を表現できていない。Noto Sans JPは汎用的すぎてブランド個性がゼロ。 |

#### P2（改善推奨 - 格を上げるために必要）

| # | 問題 | 根拠 |
|---|------|------|
| 7 | **色彩設計がぼやけている** | primary-blue 6段階 + accent-gold 3段階 + surface-warm-gray 9段階 = 18色は多すぎる。ハイエンドサイトは2-3色で構成される（Makhno: 2色、GKC Architecture: 2色）。 |
| 8 | **Works（実績）セクションの見せ方が年表型** | 年表型アーカイブは網羅性は高いが、「この会社に任せたい」という感情を起こさない。3つのフィーチャードプロジェクトのtilt effectは楽しいが、プロジェクトの「凄さ」が伝わるUIではない。 |
| 9 | **Products/Services/Team/Partnersが情報羅列** | これらのセクションは「ある」だけで「魅せて」いない。Productsの4+1+2構成は複雑。Servicesの3カテゴリーカードは差別化が弱い。 |

### 1.3 良い点（維持すべきもの）

| 維持項目 | 理由 |
|----------|------|
| Astroフレームワーク選定 | 静的生成+部分ハイドレーションは最適解。パフォーマンスとSEOの両立。 |
| 依存関係の最小化（4パッケージ） | メンテナビリティが高い。追加は慎重に。 |
| `prefers-reduced-motion`対応 | アクセシビリティの基盤が既にある。 |
| IntersectionObserverベースのスクロール連動 | パフォーマンスファーストな実装。 |
| signatureTracks.mjsの6フェーズデータ構造 | 情報設計は正しい。表現を変えるだけで化ける。 |
| JSON-LDの構造化データ | SEO基盤。維持拡張する。 |
| contact formのhoneypot spam対策 | 実用的。 |
| 色のウォームトーン方向性 | 建築として正しい。ただし絞る。 |

## 3. セクション別再設計案

### 3.1 Hero — 「何だこれは」を起こす

#### 現状の問題

- パララックス背景画像 + ダークグラデーション + テキストオーバーレイ = 2018年レベルの構成
- `MOVE HEARTS.`と`心を動かせ`の間に明確なビジュアル階層がない
- CTA2つ（実績を見る / お問い合わせ）は選択負荷（Hick's Law違反）
- Trustline（サービス一覧テキスト）は読まれない位置にある
- 登録情報（一級建築士事務所等）は権威付けとして弱い表示

#### 再設計案

**案A（推奨）: 3Dプリズム・ジェネレーティブHero**

```
視覚構成:
┌──────────────────────────────────────────────────────────────┐
│ [ダークバックグラウンド #0F0F0E]                              │
│                                                              │
│  中央: ガラスプリズム（現ImmersiveEngineのプリズムを進化）     │
│  プリズム内部で微細なパーティクルが建築的なフォルムを形成      │
│  → ゆっくり回転、ポインター追従で視点が微動                  │
│                                                              │
│                                                              │
│  左下テキストブロック（画面左1/3に配置）:                     │
│                                                              │
│    MOVE                                                      │
│    HEARTS.             ← clamp(4rem, 12vw, 9rem)             │
│                           font: GT Sectra / Canela相当       │
│    心を、動かせ。       ← clamp(1.6rem, 4vw, 3rem)           │
│                           font: Noto Serif JP                │
│                                                              │
│    AI × 建築 × デザインで、                                  │
│    未来の街と暮らしをつくる。   ← 16-18px, 60%opacity        │
│                                                              │
│    [ 実績を見る → ]          ← 唯一のCTA                     │
│                                                              │
│  右下: 一級建築士事務所 東京都知事登録...  ← 10px, 30%opacity │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**なぜこの設計か:**
1. **3Dプリズムが会社のメタファー**: プリズムは光を屈折させ虹を生む。この会社は「企画〜運用」を通すことで多面的な価値を生む。プリズム=Archi-Prismaそのもの。
2. **パーティクル→建築フォルム**: AIが建築を「生成」する瞬間をビジュアライズ。言葉で説明せず視覚で「AI建築事務所」を伝える。
3. **テキスト左寄せ**: 視線は左上から始まる（F-pattern）。3D右、テキスト左の分離で認知負荷を下げる。
4. **CTA1つに絞る**: Hick's Law。選択肢が1つなら迷わない。「お問い合わせ」はサイト下部で十分。
5. **日本語を「心を、動かせ。」に修正**: 読点で呼吸を作り、句点で着地。現在の「心を動かせ」より格が上がる。

**案B（保守案）: シネマティック写真Hero**

- 実際のプロジェクト写真（最も衝撃的な1枚）をフルブリードで表示
- Ken Burns effect（超スローズーム、20秒で5%拡大）
- テキストは案Aと同じ構成
- 3Dは排除し、パフォーマンスを最優先

**最終推奨: 案A**
理由: この会社は「技術がある」ことを証明する必要がある。写真だけでは普通の設計事務所との区別がつかない。3Dプリズムは技術力の生きた証拠になる。

#### 技術仕様

```
3Dシーン:
- Three.js PerspectiveCamera(45°)
- MeshPhysicalMaterial: transmission 0.92, thickness 1.2, ior 1.5
- パーティクル: PointsMaterial, 2000-3000パーティクル
- パーティクルモーション: sin/cos wave + noise で建築的フォルム形成
- ライティング: HemisphereLight + DirectionalLight + PointLight(accent gold)
- Fog: linear(#0F0F0E, 8, 22)
- Camera: ポインター追従（現在の実装を流用、感度を0.3→0.15に下げる）
- prefers-reduced-motion: 静止状態のプリズム + 背景に最高品質の建築写真

パフォーマンス:
- Three.jsの動的import維持（現在の実装を踏襲）
- canvasのwill-change: transform
- RAF throttle（現在の実装を踏襲）
- 初回描画まで: プリズムの静的スクリーンショットをposter画像として表示
  → LCPはposter画像（<img fetchpriority="high">）で2.5秒以内を確保
  → Three.js初期化完了後にcanvasをフェードイン（opacity transition 600ms）
```

### 3.2 Signature Experience — 「6つの領域」を圧倒的に見せる

#### 現状の問題

- 左パネル（詳細）+右タブ（6フェーズ）のレイアウトがSaaSツール的
- ポインタートラッキンググラデーション（rgba(63,116,182,0.16)）は薄すぎて効果なし
- プログレスバー（scaleY）が小さすぎて存在感なし
- capabilityタグが流れるmarqueeは読めない
- 全体的に「情報を整理した」だけで「魅せて」いない

#### 再設計案

**コンセプト: 「フルスクリーン・フェーズ・スクロール」**

│  │                                                      │    │

└──────────────────────────────────────────────────────────────┘

**なぜこの設計か:**
1. **フルスクリーン1フェーズ**: 各領域に「格」を与える。タブの6分の1ではなく、画面全体を1つの領域に使う。
2. **写真+テキスト**: 「企画」と言うだけでなく、企画フェーズの実際の写真（会議、図面、都市計画図等）を見せる。「本物感」の源泉。
3. **スクロール連動**: ユーザーのペースで進む。自動サイクル3.6秒（現状）では内容を読めない。
4. **垂直ドットプログレス**: 現在地を常に把握。6フェーズ中どこにいるか一目瞭然。
5. **marquee削除**: 読めないmarqueeは不要。capabilityはリストとして静的表示。

#### 技術仕様

```
実装方式:
- CSS scroll-snap-type: y mandatory（セクション内snap）
- 各フェーズは min-height: 100svh
- IntersectionObserver（threshold: 0.5）でactive phaseを検知
- active phase変更時:
  - テキスト: opacity 0→1, translateY(20px→0), 400ms ease-out

### 4.2 余白ルール

#### 現状の問題

- Section間: py-20(80px)~py-28(112px) → ラグジュアリー基準の半分以下
- コンテナ: max-width 1200px → 十分だが左右paddingが小さい
- テキストブロック: max-width未設定の箇所あり → 行長が長すぎる

#### 推奨余白システム

```css
/* 8pxベースユニットグリッド */

/* セクション間余白（垂直） */
--space-section-lg: 200px;  /* 主要セクション間（Hero→Signature, Works→Immersive等） */

@media (max-width: 640px) {
  --space-section-lg: 120px;
  --space-section-md: 96px;
  --space-section-sm: 72px;
}

/* コンテナ */
--container-max: 1280px;    /* 1200→1280に拡大 */
--container-px-mobile: 24px;
--container-px-tablet: 48px;
--container-px-desktop: 80px;
--container-px-wide: 120px;

/* テキストブロック最大幅 */
--prose-max: 680px;  /* ≈65文字/行。可読性の黄金比 */

/* コンポーネント内余白 */
--card-padding: 32px;
--card-gap: 24px;      /* mobile */
--card-gap-lg: 32px;   /* desktop */

/* 見出しと本文の間 */

```

**ホワイトスペースの根拠**: Pracejus, Olsen & O'Guinn の研究により、余白は知覚される品質を最大300%向上させる。高級ブランドの68%が「空間的開放感」を視覚戦略のトップ3に位置づけている。

### 4.4 動きの設計原則

#### 原則1: 動きには必ず「意味」がある

| 目的 | 許可される動き | 禁止される動き |
|------|----------------|----------------|
| 要素の出現を知らせる | opacity + translateY (fade-up) | bounce, rotate, scale(0→1) |
| 状態の変化を知らせる | color/background transition | flash, shake |
| 空間の移動を知らせる | transform: translate | 3D flip, 回転 |
| ユーザーの操作に応答する | hover scale(1.02), press scale(0.98) | jelly, elastic |
| 進捗を示す | scaleX/Y, width/height | pulse, spin（ローダー以外） |

#### 原則2: 動きのタイミング規約

```css
/* Duration Scale */
--duration-instant:  100ms;   /* ボタンpress, トグル */
--duration-fast:     200ms;   /* ホバー, アクティブ状態 */
--duration-normal:   400ms;   /* 要素出現, パネル切替 */
--duration-slow:     600ms;   /* セクション遷移, 画像クロスフェード */
--duration-dramatic: 800ms;   /* Hero初回ロード, 3Dトランジション */

/* Easing */
--ease-out:     cubic-bezier(0.16, 1, 0.3, 1);  /* 要素出現（現在と同じ） */
--ease-in-out:  cubic-bezier(0.65, 0, 0.35, 1);  /* 状態変化 */
--ease-spring:  cubic-bezier(0.34, 1.56, 0.64, 1); /* インタラクティブ要素のみ */

/* Stagger（複数要素の連続出現） */
--stagger-fast: 50ms;   /* ナビ項目 */
--stagger-normal: 100ms; /* カードリスト */
--stagger-slow: 150ms;  /* ギャラリー */
```

#### 原則3: `prefers-reduced-motion` の徹底

```css
/* "No-motion-first" アプローチ（Tatiana Mac推奨） */

/* ベース: 動きなし */
[data-reveal] {
  opacity: 1;
  transform: none;
}

/* 動き許可ユーザーのみアニメーション適用 */
@media (prefers-reduced-motion: no-preference) {
  .motion-ready [data-reveal] {
    opacity: 0;
    transform: translate3d(0, 30px, 0);
    transition: var(--duration-normal) var(--ease-out);
  }
  .motion-ready [data-reveal].in-view {
    opacity: 1;
    transform: none;
  }
}
```

```javascript
// Three.js: reduced-motion時は静的レンダリング
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)');

if (prefersReducedMotion.matches) {
  // Three.jsをロードしない
  // 代わりに高品質な静的画像を表示
  showStaticFallback();
} else {
  const THREE = await import('three');
  initScene(THREE);
}
```

## 5. 批判役レビュー体制

### 5.1 三役の定義と否決条件

#### Brand Critic（ブランド）

**役割**: このサイトが「世界トップクラスのAI建築設計事務所」として信頼されるかを判定

**否決条件（1つでも該当したら差し戻し）:**

| ID | 否決条件 | 検証方法 |
|----|----------|----------|
| BC-1 | Hero を見て3秒以内に「AI×建築の会社」だと認識できない | 5人にHeroスクリーンショットを3秒見せ、「何の会社か」を回答させる。3/5が正解しなければNG。 |
| BC-2 | フォント・色・余白のいずれかが「テンプレ感」を出している | 無料テンプレート（ThemeForest等）と並べて区別がつかなければNG。 |
| BC-3 | 「技術がある」「実績がある」「センスがある」の3点が同時に伝わらない | サイト全体を60秒閲覧後、「この会社の強みは？」を自由記述。3点中2点以上が出なければNG。 |
| BC-4 | 競合他社サイト（日建設計、隈研吾等）と比較して「格負け」している | 主観だが複数名で判定。1名でも「負けている」と言えばNG。 |

#### UX Critic（体験）

**役割**: ユーザーが迷わず、疲れず、目的（問い合わせ）に到達できるかを判定

**否決条件:**

| ID | 否決条件 | 検証方法 |
|----|----------|----------|
| UX-1 | 初訪問ユーザーがContactに到達するまでに3回以上迷う | 5人にタスク「この会社に問い合わせてください」を指示。3回以上「戻る/迷う」行動があればNG。 |
| UX-2 | モバイルでCTAが画面内に見えない状態が5秒以上続く | iPhoneSE〜iPhone15Proで全セクション確認。CTAが5秒以上不可視の区間があればNG。 |
| UX-3 | 1セクション内の選択肢が5つを超える | Hick's Law違反。ナビ、タブ、カードいずれも5つ以内。 |
| UX-4 | ページ全体のスクロール量がデスクトップで8000pxを超える | 情報過多。8000px以内に全セクションが収まること。 |
| UX-5 | Lighthouse Accessibilityスコアが95未満 | 自動テスト。 |

#### Motion Critic（動き）

**役割**: すべての動きが「目的」を持ち、「意味なく動いている」要素がゼロであることを保証

**否決条件:**

| ID | 否決条件 | 検証方法 |
|----|----------|----------|
| MC-1 | 動きを見て「なぜ動いているか」を1文で説明できない | 全アニメーション要素をリスト化。各要素の動きの目的を1文で書けなければNG。 |
| MC-2 | `prefers-reduced-motion: reduce` でコンテンツが欠落する | 全セクションを reduced-motion で確認。情報が欠けたらNG。 |
| MC-3 | 60fps を維持できないアニメーションがある | Chrome DevTools Performance パネルで計測。ドロップフレームがあればNG。 |
| MC-4 | アニメーション完了を待たないと次の操作ができない | 全トランジション中にクリック/スクロール操作を試行。ブロックされたらNG。 |
| MC-5 | ページロード後、ユーザー操作なしで3つ以上の要素が同時に動いている | 自動再生アニメーションの棚卸し。3つ以上同時はNG（Heroの3Dは1要素とカウント）。 |

## 7. リスクと回避策

| # | リスク | 影響度 | 発生確率 | 回避策 |
|---|--------|--------|----------|--------|
| R-1 | **Three.js強化でLCPが3秒超え** | 高 | 高 | poster画像をLCP要素にし、Three.jsは非同期で後からフェードイン。canvas自体はLCP対象外にする。 |
| R-2 | **フォント変更で日本語レンダリングが崩れる** | 中 | 中 | Noto Serif JP は Google Fonts で最も安定した明朝体。変更前にmobile Safari/Android Chromeで表示確認。問題があればNoto Sans JP維持+ウェイト変更のみで対処。 |
| R-3 | **scroll-snap がiOS Safariで不安定** | 中 | 低 | scroll-snap-type: y proximityで代替。完全snapが不要ならIntersectionObserverによるJS制御にフォールバック。 |
| R-4 | **セクション削減で既存SEOランキングが低下** | 中 | 低 | 削除セクションのコンテンツは下層ページに移動（/services, /partners等）。301リダイレクトで既存URLを維持。内部リンク構造を保全。 |
| R-5 | **Canela/GTSectra等の有料フォントのライセンス問題** | 低 | 高 | 推奨案にPlayfair Display（無料）を併記済。無料フォントで80%の品質は出る。 |
| R-6 | **ドメイン別3Dシーン（Phase 2）の工数爆発** | 高 | 中 | Phase 2の3D強化は段階的に。まずStep 2-2（カメラ/発光強化のみ）を先行。ドメイン別オブジェクトは追加Phase（Phase 2.5）として分離可能。 |
| R-7 | **ゴールドカラー削除に対するクライアント反発** | 中 | 中 | 「ゴールドを使わない」ではなく「写真の中でゴールドが活きる」と説明。UI上のゴールドは散漫さの原因。建築写真の中にゴールド/ウッドの温かみは十分存在する。 |
| R-8 | **Astroのis:inline制約でGSAPが使えない** | 中 | 低 | 現在の実装はバニラJSのみでアニメーションを実現済。GSAPが必要な場合はscript tagでCDN読み込み or astro.config.mjsでvite integrationを追加。ただし依存関係最小化の方針を維持し、CSS/バニラJSで実現できるものはそちらを優先。 |

### 技術根拠

| 技術判断 | 根拠 |
|----------|------|
| Astro維持（Next.js移行しない） | 4依存パッケージの軽量性は資産。SSGで十分なサイト規模。移行コストに対してリターンが不明確。 |
| Three.js維持（R3F移行しない） | ReactがないのでR3Fは不適。バニラThree.jsで既に実装基盤がある。 |
| GSAP導入しない | 現在のCSS Transitions + IntersectionObserverで大半の動きは実現可能。依存最小化方針を維持。必要になったらPhase 2で再検討。 |
| WebGPU対応しない（現時点） | 2025年時点のブラウザサポートは限定的。WebGLで十分。将来のPhaseで検討。 |

## 最終確認

この計画が成功した場合、サイト訪問者は以下の感情を順に体験する:

1. **「何だこれは」**（Hero, 0-3秒）
   → 3Dプリズム + パーティクル生成 + 大タイポの組み合わせが、見たことのない第一印象を作る

2. **「この会社、普通じゃない」**（Signature, 3-20秒）
   → 企画から運用まで6領域を横断する事実が、フルスクリーンの写真と共に突きつけられる

3. **「実績も技術も本物だ」**（Works + Immersive, 20-45秒）
   → 実際のプロジェクト写真が非対称グリッドで格を持って並び、3Dインタラクティブが技術力を証明する

4. **「任せたい」**（Contact到達, 45-60秒）
   → 全ての情報が「この会社に依頼する」という決断に自然と導く

**この感情が起きない設計は不採用。**
